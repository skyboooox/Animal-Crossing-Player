# 概述

本文件是仓库级工作指令，用于约束编码智能体在当前仓库中的默认行为。

目标是提供高信号、可执行、低歧义的约束：说明哪些事情不能从代码里安全推断，哪些验证必须完成，哪些边界不能越过。不要把本文件当作完整工程手册，也不要把可从代码或配置中直接读取的信息重复堆进来。


## 用途与作用域

- 本文件默认作用于所在目录及其子目录。
- 如果某个子目录有独立语言、框架、测试方式或安全边界，应在该子目录增加更具体的指令文件，而不是把所有细节塞进根目录文件。
- 保持本文件短、具体、可维护；内容过长时优先拆分到子目录。
- 用户当前对话中的直接要求优先级高于本文件；系统、开发者和安全策略优先级也高于本文件。
- 如果本文件与仓库真实代码、配置或用户最新要求冲突，应先指出冲突并按更高优先级信息处理。

## 事实来源

- 开始任务先读真实仓库：结构、入口、公开接口、命令、测试、生成文件、禁止修改区。
- 命令优先来自 `README`、包管理配置、CI、构建脚本、Makefile、任务配置。
- 公开 API、文档入口、迁移、生成物、忽略规则优先来自导出文件、配置、`.gitignore`、目录约定。
- 不能从仓库确认的事实不要编造。只有产品意图、兼容性边界、交付形式不清楚时再问用户。
- 若任务要求参考官方方式、当前最佳实践、网络信息，或涉及易变 API、依赖版本、安全公告、法规、价格、模型，先联网核实。

## 工作规则

- 编辑前检查文件系统状态和 `git status`。
- 保护已有未提交改动。不要覆盖用户或其他智能体的工作。
- 修改聚焦当前任务。不顺手重命名、搬迁、格式化或清理无关内容。
- 遵循现有模式：目录、命名、错误处理、测试组织、风格。
- 不新增生产依赖，除非任务明确需要且收益明显。新增前说明用途、替代方案、影响面。
- 不把可由代码或配置读取的事实硬编码进文档。必须写入时，与真实来源一致。
- 需求、范围、兼容性或交付形式不清楚时，先读代码消除可发现不确定性。仍有高影响歧义再问。
- 用户限定写入范围时，严格只改授权文件。

## 全场景精简与 token 预算

- 默认省 token：代码、注释、回复、文档、计划、交接都去掉寒暄、填充词、弱化语、重复表述。
- 保留全部技术事实、约束、风险、验收标准。短句可以，含义不能缩水。
- 精简自然语言，不改代码语义、公开接口、错误信息、协议字段、配置键。
- 代码优先减少样板、重复分支、无效抽象、无意义注释；不要为省字牺牲可读性、类型安全、错误处理。
- 回复用户先给结论、变更、验证、风险；避免长铺垫和重复复述。
- 文档只写会改变行为的规则、真实命令、真实路径、必要背景；删除可从代码直接读取的堆叠事实。
- 精确保留代码块、行内代码、URL、Markdown 链接、文件路径、命令、环境变量、API 名、库名、专有名词、日期、版本号、数字。
- 保留 Markdown 结构：标题、列表层级、表格结构、前置元数据。
- 多个示例表达同一模式时，只留一个代表例。
- 安全警告、不可逆操作确认、复杂步骤、用户困惑时，优先清晰，不强行压缩。
- 不引入外部压缩工具或插件依赖，除非用户明确要求。

## 验证要求

- 代码改动先跑最相关的最小验证，再按风险补充更广检查。
- 改公开 API、导出类型、CLI 参数、配置项、网络协议时，同步验证运行时行为、类型声明、文档示例。
- 改 UI 或用户可见流程时，做真实运行或等效端到端验证。适合浏览器验证时优先用 Playwright 或实际浏览器。
- 改数据库、迁移、队列、缓存、权限、部署、CI 时，不只跑单元测试；还要查配置、脚本、回滚风险。
- 纯文档改动通常不跑完整测试，但要自查命令、路径、示例是否真实。
- 无法执行的检查，最终说明写明原因、未验证内容、建议后续验证。

## 文档与 API 同步

- 中英文文档、README、示例、类型声明、实际导出 API 必须一致。
- 示例短小可运行，使用当前真实导出名称、默认值、路径。
- 改公开行为时，同步检查公开接口、README、示例、类型声明、测试。
- 不允许一个 README 更新，另一个仍保留旧选项名、旧默认值、旧示例地址。
- 若存在 `PROJECT.md`、`PROMPT.md`、设计文档或产品说明，把它们作为意图来源。缺失不阻塞普通维护；重大产品或架构决策前提醒用户确认。

## PLAN.md

- 当前任务对应 `PLAN.md` 阶段、步骤、待办时，完成后更新对应状态。
- 执行中发现会影响计划的新问题、约束变化、路径调整、新任务时，同步更新 `PLAN.md`，或在最终说明建议更新。
- 不默认创建、移动、归档、重写 `PLAN.md`，除非用户明确要求或仓库已有约定。
- 若 `PLAN.md` 仍有未实现、未验证、未关闭内容，任务结束时提示剩余项。
- `PLAN.md` 全部完成后，只有用户要求或仓库约定要求时才归档，并标明“已完成”或“已废弃”。

## 多智能体模型

复杂任务可用规划者、生成者、评估者三角色。角色表示工作意图，不要求存在同名工具。无可用子智能体时，主智能体承担全部职责。

### SubAgent 使用

- 只在任务可并行、边界清晰、收益大于上下文成本时使用 SubAgent。
- 阻塞主路径、强耦合、高歧义任务优先由主智能体直接处理。
- 每个 SubAgent 只给一个目标、明确文件/模块所有权、范围外内容、输出格式、验证要求。
- 并行 SubAgent 必须写集成约束：不得回滚他人改动，不得编辑未授权文件，不得跨模块顺手清理。
- 模型选择优先性价比：搜索、静态阅读、简单验证、文档草拟用低成本/快速模型；复杂架构、安全、跨模块实现、最终评审才用更强模型。
- 主智能体负责复核 SubAgent 结果、整合冲突、运行最终验证、对交付质量负责。

### SubAgent 的模型约束

- 编码工作的时候,必须要使用gpt-5.3-codex-spark模型,思考等级超高
- 进行测试,获取信息,整理文档工作的时候,必须要使用gpt-5.4-mini,思考等级高

### 规划者

- 将模糊需求变成可实施规格和技术方案。
- 明确目标、范围、非目标、交付物、验收标准、风险。
- 技术栈、模块、数据模型、顺序优先适配现有架构。
- 大型工作拆成 Sprint，并写清输入、输出、验证。

### 生成者

- 每次只实施边界清晰的 Sprint。
- 变更局部、可评审、可回滚。
- 功能、重构、格式化、无关清理不要混在一起。
- 并行工作时只改自己负责的文件或模块，不回滚他人改动。

### 评估者

- 作为 QA、回归测试、代码评审门控。
- 按 Sprint 契约验证用户可见行为、边界条件、失败路径。
- 产品表面层级合适时，用 Playwright 或等效运行时验证真实交互。
- 失败来自产品代码时，给出最小复现、实际结果、期望结果，再返回修正。

### 编排

- 主智能体负责规划、委派、集成、冲突处理、最终交付。
- 任务可分离且工具支持时，可并行委派规划、实施、测试、评审。
- 避免多个智能体同时编辑同一文件或模块。无法避免时，先明确所有权和合并顺序。
- 微小、强耦合、上下文敏感任务由主智能体直接完成。

## Sprint 契约

实施 Sprint 前定义轻量契约，供生成者和评估者共享：

- 确切目标
- 范围内文件或模块
- 范围外内容
- 完成定义
- 必要验证步骤
- 通过/失败阈值
- 已知风险和回滚方式

评估者无法验证，或验证不达阈值，则 Sprint 未完成，返回修正。

## 上下文交接

交接时丢弃陈旧对话细节，只保留下一步所需状态。优先短结构，不写长叙述。

```json
{

  "scope": [],
  "constraints": [],
  "definition_of_done": [],
  "files": [],
  "status": "",
  "open_questions": [],
  "risks": [],
  "verification": []
}
```

## 四层架构标准

新项目、重大重构、缺少一致架构的模块，默认采用集中编排、底层独立的四层结构。已有成熟一致架构时保留现状，不为套结构机械重命名、搬迁、拆分稳定代码。

### 原则

- 业务流程控制中心化。
- 底层简单、可复用、轻依赖。
- 优先浅调用链，避免多层间接。
- 流程、状态机、重试、错误处理、跨领域决策放在可读核心层，不藏进工具函数。

### L1 Entry

- 接收 HTTP、CLI、队列、定时任务、UI 事件等外部输入。
- 解析并验证请求格式。
- 转发给 L2。
- 不放业务决策、跨领域编排、持久化细节。

### L2 Core

- 拥有业务流程和决策逻辑。
- 协调 L3 和 L4。
- 包含分支、重试、错误处理、权限判断、工作流规则。
- 读起来应像流程或状态机。

### L3 Business

- 实现业务能力单元。
- 将 L4 原子能力组合成领域动作。
- 保持被动：执行工作，不拥有跨领域编排权。
- 无明确理由和许可时，避免导入同级 L3 模块。

### L4 Atom

- 提供微小、可复用、底层能力。
- 仅依赖标准库、平台 API、广泛适用第三方库。
- 不从 L1、L2、L3 导入项目特定代码。
- 函数功能单一、精简。

### 依赖

- L1 可调用 L2。
- L2 可调用 L3 和 L4。
- L3 可调用 L4。
- L4 不调用项目层级。
- 避免 L3 到 L3 横向调用。
- 不把会话、权限、工作流、状态机决策藏在 L3 或 L4。

```text
/project_root
  /L1_Entry
  /L2_Core
  /L3_Business
  /L4_Atom
```

## 调试规则

Bug 从工作流顶部向下查：

1. 先查 L2：编排、分支、状态、错误处理。
2. 流程正确后，查产生错误结果的 L3 或 L4。
3. 编排 Bug 在 L2 修；实现 Bug 在对应 L3 或 L4 修。

目标：心智路径保持浅层，通常是 `L2 -> 具体目标`。

## 安全护栏

- 不提交密钥、令牌、私钥、生产密码、个人敏感信息。
- 不静默扩大文件权限、网络暴露面、CORS 范围、数据库权限、云资源权限。
- 未获授权不执行破坏性命令：删除数据、重置历史、强推、清空缓存、重建数据库、覆盖迁移。
- 不静默修改 CI、部署、生产配置、数据库迁移、安全策略；必要时先说明影响面和回滚方式。
- 不把测试后门、调试端点、日志泄露、临时绕过留在生产路径。
- 用户明确要求保留本地便利凭据时，按本次要求处理；不要扩展到其他仓库或生产环境。

## 维护原则

- 构建、测试、发布、安全、数据处理流程变化时，同步更新本文件。
- 每条规则都应减少已知错误、消除高影响歧义、节省重复探索。
- 泛泛而谈、所有仓库都适用、不会改变智能体行为的规则应删除或移到团队文档。
